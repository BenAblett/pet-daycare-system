---Evaluation--
The AI provided tests that were correct a vast majority of the time, for the small test classes such as parrot and cat, the AI was effective in quickly generating test cases that were accurate and useful.
But when the AI had to deal to with PetsDayCareAPITest there was major difficulty in achieving coverage as it ended up generating redundant classes that had to be removed.
Initially it was missing tests that checked edge cases, null, and mixed values, eventually this was resolved upon iteration.
Some tests did fail but this was generally not due to the test being poor but my methods being incorrectly designed, in doing so the AI effectively revealed many bugs in my design.
If a test failed due to the AI it was likely due to the AI using an incorrect value for the test such as using "Lab" instead of "Labrador".

2 strengths of the AI-generated tests:
-Quick test generation

-Useful for quick debugging of failures in the test

2 weaknesses / issues:
-Had trouble with large class where tests began to become redundant

-Missed edge case, boundary values, null tests and, mixed value tests initially

-Occasionally used incorrect expectation

---Improvements Made---
AI used incorrect value such as "Lab" which I replaced with "Labrador"
AI suggested additional tests, but when they didn’t improve coverage, I instead analysed the code manually and targeted missing branches.

---Final Judgement---
I would not trust purely AI-generated tests in a real project. 
The tests it provided were accurate and effective, but it struggled with coverage, initially the tests had a coverage of 76% before manual intervention.
Without manual intervention and overlook I would not be confident in strong coverage tests that adhere to the specs.

Risks of relying on AI include:

Generating redundant tests that do not improve coverage
Incorrect assumptions about how the code works
Missing important edge cases (e.g., null handling)
